# Lvau 0.4.0 hardening design

Date: 2026-07-15

## Decision

Lvau 0.4.0 introduces capsule format version 2 for newly encrypted files. Readers continue to accept version 1 capsules, including the six-field envelope written by Lvau 0.2.x. Version 1 is compatibility-only and is reported as legacy because its payload does not authenticate the declared plaintext length.

This is a minor SemVer release: the public CLI remains source-compatible where practical, existing capsules remain readable, and the only intentional behavior change is that new output uses a safer on-disk format. A 0.3.1 patch cannot fix the truncation issue without changing the format.

## Security boundary

Version 2 derives the payload AAD commitment from a domain-separated, canonical postcard encoding of:

- the complete envelope header;
- the declared plaintext length;
- the primary and secondary nonces;
- public metadata;
- content type and public label; and
- the policy-override flag.

Mutable attestations (author signature, approval signatures, release metadata, and recovery metadata) remain outside the payload commitment so they can be attached without re-encrypting the payload. They must not be treated as authenticated merely because they are present. Signature verification remains a separate operation.

Every version 2 capsule containing an empty plaintext carries an authenticated empty payload frame. All version 2 decrypt and verify paths reject trailing bytes after the expected payload. Together with the committed plaintext length, this prevents prefix truncation, empty-payload forgery, and unauthenticated suffixes.

No new cipher or primitive is introduced. Lvau continues to use the existing AEAD algorithms and changes only what is supplied as their associated data.

## Compatibility reader

A single bounded capsule parser owns envelope framing. It:

1. reads the four-byte little-endian envelope length;
2. rejects zero-length and envelopes larger than 1 MiB before allocation;
3. requires an exact postcard decode with no unused bytes;
4. first decodes the current envelope, then the exact Lvau 0.2.x six-field legacy envelope;
5. converts legacy fields to the current in-memory model; and
6. validates magic, supported version, algorithms, recipients, KDF parameters, and the version-specific AAD commitment.

CLI inspect/policy, preflight, signing, release metadata, bundle inspection, decryption, and verification use this parser instead of duplicating unchecked slicing.

## KDF and resource limits

Password capsules must use the Argon2id tuple defined by their security profile. This preserves every tuple emitted by released Lvau versions while rejecting attacker-selected memory, iteration, and parallelism values before Argon2 allocation. Key-pair capsules must not carry password KDF parameters. Writer-side envelope length and recipient counts are checked before output.

## File safety

Encryption and decryption write through same-directory, randomly named RAII temporary files. Failure removes partial plaintext automatically. On Unix, rename replaces the destination atomically and the parent directory is synced; temporary plaintext and key material are created with owner-only permissions. Windows keeps a documented replacement fallback where the platform API cannot replace an open destination portably.

Private signing keys, hybrid private keys, and recovery shares are written atomically with owner-only Unix permissions. Windows ACL failures are surfaced rather than silently weakening protection.

## Bundle validation

Bundle manifests are validated completely before extraction:

- checked integer conversions and additions;
- range and overlap checks;
- normalized, collision-free relative paths;
- file-type restrictions at pack time; and
- BLAKE3 verification of every entry.

`verify-bundle` performs the same manifest and entry validation as extraction and does not report success after checking only the outer AEAD.

## Signatures and policy claims

For version 2, signer metadata is included in the author-signature statement and the actual verification-key fingerprint must match the stored fingerprint. Approval verification recomputes the artifact commitment before accepting a seal. Documentation and preflight output explicitly distinguish presence from cryptographic verification.

The larger attestation-chain redesign (portable trust stores, certificate paths, and signed policy evaluation) is deferred. Lvau 0.4.0 will not claim those guarantees.

## Release and repository layout

Canonical ownership is `latteworkspace/lvau`. The Rust workspace is the source for CLI/GUI binaries. `lattes.jp` remains the public website, and `lvau-api` remains the OCI-facing service. The repositories share version/download/API contracts but are not folded back into one workspace.

Release workflows build once per target, test before packaging, validate tag-to-version consistency, create SHA-256 checksums, publish an SBOM and GitHub artifact attestation, and grant write permissions only to the release job. This work prepares workflows only; it does not create a tag, release, push, or deployment.
